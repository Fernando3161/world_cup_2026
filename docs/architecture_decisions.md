# World Cup Knockout Forecast Website — Architecture Decisions

## 1. Purpose

This document defines the main architecture decisions for the World Cup knockout forecast website.

The goal is to give coding agents clear boundaries before implementation begins. The project must remain professional, testable, reproducible, transparent, and cheap to deploy.

The application should feel like a polished public web product, but it must not require paid server infrastructure for normal user interaction.

---

## 2. Core Architecture Decision

The application will be a static React + TypeScript website generated from local data snapshots.

Python will be used only for build-time, setup-time, data-processing, and validation tasks.

The deployed public website must not require a live Python backend.

All normal user interactions must run client-side in the browser.

This includes:

- Bracket display
- Match probability display
- Manual result overrides
- Downstream bracket recalculation
- Baseline forecast
- Current scenario forecast
- Champion probability recalculation
- Model toggle, once the historically informed model exists

---

## 3. Deployment Target

The primary deployment target is GitHub Pages or an equivalent static-site host.

The deployed website must consist only of static frontend assets:

- HTML
- CSS
- JavaScript
- Static JSON files
- Static image or flag assets

The deployed website must not depend on:

- A running Python server
- A backend database
- Server-side sessions
- User authentication
- Runtime scraping
- Runtime calls to rating sources


---

## 4. GitHub Pages and CI/CD Architecture

The deployment target is GitHub Pages using GitHub Actions as the publishing source.

GitHub Pages is a static-site hosting target. Therefore, the production deployment must publish only static files generated by the frontend build process. GitHub Actions should build, validate, package, and deploy those files.

The project must use the official GitHub Pages deployment flow unless explicitly revised:

```text
actions/checkout
↓
Set up Python
↓
Install Python dependencies
↓
Validate source data
↓
Generate frontend JSON
↓
Set up Node
↓
npm ci
↓
npm run lint
↓
npm run typecheck
↓
npm test
↓
npm run build
↓
actions/configure-pages
↓
actions/upload-pages-artifact
↓
actions/deploy-pages
```

### 4.1 Required Workflows

The repository should contain at least two GitHub Actions workflows.

```text
.github/workflows/ci.yml
.github/workflows/deploy-pages.yml
```

A future optional manual workflow may be added for data refresh:

```text
.github/workflows/refresh-data.yml
```

The optional data refresh workflow must not be part of the normal deployment path unless explicitly approved.

### 4.2 CI Workflow

`ci.yml` should run on pull requests and pushes to development branches.

Its purpose is to verify that the project is safe to merge.

The CI workflow should run:

- Python dependency installation
- Python tests
- Data validation
- Frontend data generation
- Node dependency installation using `npm ci`
- Frontend linting
- TypeScript checking
- Frontend tests
- Static frontend build

The CI workflow must not deploy to GitHub Pages.

### 4.3 Deployment Workflow

`deploy-pages.yml` should run on pushes to `main` and by manual dispatch.

Deployment must happen only after validation and tests pass.

The deployment workflow should:

- Check out the repository
- Set up Python
- Install Python dependencies
- Validate data
- Generate frontend static JSON
- Set up Node
- Install frontend dependencies with `npm ci`
- Run linting, type checking, and tests
- Build the Vite frontend
- Upload the generated static site as a Pages artifact
- Deploy the artifact to GitHub Pages

The deployment workflow must use minimal permissions:

```yaml
permissions:
  contents: read
  pages: write
  id-token: write
```

The deployment workflow should use a Pages deployment concurrency group so multiple deployments do not race.

### 4.4 GitHub Pages Build Output

The frontend build output should be the Vite `dist` directory.

The Pages artifact should upload only the built static site, not the whole repository.

Recommended artifact path:

```text
frontend/dist
```

The build must ensure that generated data is available in the built site, for example:

```text
frontend/public/data/tournament.json
```

or an equivalent public static path copied before `npm run build`.

### 4.5 Vite Base Path

The Vite configuration must support GitHub Pages project-site deployment.

If the site is deployed at:

```text
https://<owner>.github.io/<repository-name>/
```

then Vite must use the correct base path:

```text
/<repository-name>/
```

If a custom domain or user site is used later, the base path may be changed accordingly.

The deployment workflow should make the base path explicit through configuration or environment variables. Do not rely on accidental local defaults.

### 4.6 Static Routing

If the React app uses client-side routing, the app must be compatible with GitHub Pages refresh and deep-link behavior.

Acceptable options are:

- Avoid deep client-side routes in the MVP.
- Use hash-based routing.
- Generate an appropriate `404.html` fallback for GitHub Pages.

The MVP should avoid unnecessary routing complexity.

### 4.7 No Secrets in Deployment

Deployment to GitHub Pages must not require private API keys or secrets.

The default `GITHUB_TOKEN` provided by GitHub Actions is sufficient for Pages deployment.

Runtime frontend code must not include secrets.

### 4.8 No Runtime Data Fetching from External Sources

Deployment may validate and package committed local snapshots.

The deployed frontend must not fetch live Elo data, live match data, or historical databases at runtime.

Data acquisition from external sources must be a separate manual or setup step. It must produce local snapshot files that can be reviewed and committed.

### 4.9 GitHub Pages Limits

The project must remain within GitHub Pages constraints.

The site should keep static assets small and avoid shipping unnecessary historical datasets to the browser.

Large historical results, calibration data, and raw data should stay outside the deployed frontend bundle unless specifically needed by the UI.

### 4.10 Branch and Deployment Policy

The recommended branch policy is:

- Pull requests run CI only.
- Pushes to `main` run CI and deploy to GitHub Pages.
- Manual dispatch can be used for controlled redeployment.
- Official-result updates during the tournament should be made through commits to static data files followed by the normal deployment workflow.

The production GitHub Pages site should reflect the content of the `main` branch plus generated build artifacts.

## 6. Frontend Technology

The frontend will use:

- React
- TypeScript
- Vite
- Custom CSS

No heavy UI framework should be introduced in the MVP unless there is a strong reason.

The frontend should be professional, responsive, accessible, and clear.

The UI should prioritize:

- A readable knockout bracket
- Clear team names
- Clear flags
- Clear probabilities
- Clear distinction between predicted winners, user overrides, and later official locked results
- Clear distinction between the simple Elo model and the historically informed Elo model once both are available


### 6.1 Visual Design Decision

The public website previously used Visual Design v1: an editorial, data-rich visual style inspired by the Financial Times.

Visual Design v1 meant the product should feel like a serious newspaper-style data interactive:

- Warm paper-like page background
- Restrained colour palette
- Strong editorial typography
- Fine rules and borders
- Compact but readable data hierarchy
- Calm interaction states
- No sportsbook or gaming visual language

Visual Design v1 is now superseded by Visual Design v2 for the MVP.

Visual Design v2 is:

```text
premium football analytics dashboard
```

The rationale for this change is that the v1 FT-inspired aesthetic was too restrained for the product. The MVP is functionally good, but the old design felt too empty, quiet, and unattractive. The bracket is the product and must become the immediate visual hero.

The v2 product should use:

- Light page shell
- Dark bracket panel or canvas
- Bracket-first first viewport
- Stronger but controlled color system
- Polished knockout-tree presentation
- Professional analytical tone
- No betting-site aesthetic
- No FIFA clone
- No childish sports-broadcast aesthetic

The bracket must be visible prominently in the first desktop viewport. The page must avoid large empty ceremonial whitespace before the bracket.

Stage 6.3 refines this into Visual Design v2.1:

- The bracket appears immediately after the title/header area.
- Model controls, scenario status, reset controls, champion summary, and explanatory content sit below the bracket.
- The landing view must feel like a professional knockout bracket product, not a dashboard where the main visual object is delayed by secondary panels.

The implementation must not copy the Financial Times brand identity or any football governing-body identity.

The project must not use:

- The FT logo
- Financial Times trademarks as product branding
- Proprietary FT fonts unless separately licensed
- FT articles, headlines, imagery, or content
- Design choices that imply endorsement or affiliation

The previous design target was:

```text
FT-inspired editorial forecast aesthetic
```

The current design target is:

```text
premium football analytics dashboard
```

not:

```text
Financial Times clone
FT-branded product
FIFA clone
betting product
```

The visual design contract is defined in:

```text
docs/visual_design.md
```

Frontend CSS should be implemented through design tokens, preferably in:

```text
frontend/src/styles/tokens.css
frontend/src/styles/global.css
frontend/src/styles/typography.css
frontend/src/styles/layout.css
frontend/src/styles/components.css
```

Do not use a heavy UI framework just to achieve this style.

Do not hard-code visual constants inside React components.

## 7. Backend Decision

There will be no runtime backend in the MVP.

Python is not part of the deployed runtime application.

Python will be used for:

- Fetching or preparing current Elo data
- Fetching or preparing historical match result data
- Creating local CSV snapshots
- Cleaning country/team names
- Mapping country aliases
- Validating ratings
- Reconstructing historical Elo time series
- Calibrating historical rating-difference models
- Generating frontend-ready JSON
- Running reference model calculations
- Running tests for probability logic
- Optional Monte Carlo validation

The website must continue to work after deployment without any Python process running.

## 8. Data Source Decision: Current Elo Ratings

### 7.1 Primary Source

The primary rating source for the Elo-based model should be:

```text
World Football Elo Ratings
https://www.eloratings.net/
```

This is the preferred source for national-team Elo-style ratings.

Important note:

- There is no single official FIFA "World Football Elo" feed.
- FIFA publishes official ranking points, and the current FIFA ranking method is Elo-like, but FIFA ranking points and World Football Elo ratings are not the same source.
- The project should call its model "Elo-based" or "rating-based" unless the exact source is World Football Elo.

### 7.2 Accessible Mirror / Fallback Source

If direct extraction from `eloratings.net` is technically unreliable, an accessible fallback may be:

```text
Football Elo Ratings
https://www.footballratings.org/
```

This site states that its Elo data is sourced from `eloratings.net` and updated nightly. It can be used as a technical fallback or cross-check, provided the source metadata records that the underlying ratings come from `eloratings.net`.

### 7.3 Pre-WC2026 Snapshot Rule

For a pre-World-Cup forecast, the project must use a frozen rating snapshot from immediately before the tournament starts.

Recommended snapshot definition:

```text
rating_snapshot_kind = "pre_tournament"
rating_snapshot_date = "2026-06-10"
```

or the latest available rating table before the first World Cup 2026 match.

Do not use an Elo table updated after World Cup matches have already started if the purpose is to produce a pre-tournament forecast.

For a forecast made after the Round of 32 is known, the project may instead use:

```text
rating_snapshot_kind = "pre_knockout"
```

This must be explicitly recorded, because pre-tournament and pre-knockout forecasts answer different questions.

### 7.4 Local Snapshot Requirement

The runtime app must not query Elo sources.

The setup/build process must create a local snapshot:

```text
data/snapshots/ratings.csv
```

This file becomes the authoritative rating input for the frontend build.

## 9. Data Source Decision: Historical Results and Historical Elo

### 8.1 Primary Historical Match Results Source

The primary historical match results source should be:

```text
Mart Jürisoo — international_results
https://github.com/martj42/international_results
```

This source provides CSV files for men's full international football results, including:

- `results.csv`
- `shootouts.csv`
- `goalscorers.csv`
- `former_names.csv`

The project should use at least `results.csv` and `shootouts.csv`.

The dataset is suitable for building a historically informed rating-difference calibration layer.

### 8.2 Kaggle Mirror

The Kaggle version may be used for manual download or convenience:

```text
International football results from 1872 to 2026
https://www.kaggle.com/datasets/martj42/international-football-results-from-1872-to-2017
```

The GitHub repository should be preferred for reproducible automated fetching.

### 8.3 OpenFootball Mirror

The OpenFootball mirror may be useful if tournament-by-tournament text files are convenient:

```text
openfootball/internationals
https://github.com/openfootball/internationals
```

This should be treated as a mirror/secondary format, not the primary project source.

### 8.4 Historical Elo Database Strategy

The project should not depend on a third-party historical Elo table as the primary source of truth.

Instead, the preferred strategy is:

```text
historical match results
        ↓
local Elo reconstruction script
        ↓
historical_elo_timeseries.csv
        ↓
historical_match_features.csv
        ↓
historical calibration model
```

This gives the project control over:

- Starting ratings
- Elo formula version
- Match weighting
- Treatment of draws
- Treatment of extra time
- Treatment of penalty shootouts
- Neutral/home adjustment
- Recency filters
- Training/test splits

Third-party historical Elo datasets may be used only as secondary references or sanity checks.

## 10. Data Flow

The intended data flow is:

```text
External current Elo source
        ↓
Python fetch/preparation script
        ↓
Local current Elo CSV snapshot
        ↓
Python validation and transformation
        ↓
Frontend-ready static JSON
        ↓
React app
        ↓
Browser-side bracket and probability calculations
```

For the historically informed model, the additional data flow is:

```text
External historical results source
        ↓
Python fetch/preparation script
        ↓
Historical results CSV snapshot
        ↓
Local Elo reconstruction
        ↓
Historical match feature table
        ↓
Calibration model
        ↓
Frontend-ready static model parameters
        ↓
React model toggle and browser-side forecast
```

The public website must consume prepared static data, not live external data.

## 11. Rating Data

The first version will use a local CSV rating snapshot.

The rating CSV should contain at least:

- Team ID
- Display name
- Country code
- Rating value
- Rating source
- Rating snapshot kind
- Rating date
- Retrieval date
- Source URL or source identifier
- Source license or usage note, if available

If the rating CSV is missing, the setup or build process must fail with a clear error message.

The frontend must not attempt to fetch live rating data from the internet during user interaction.

---

## 12. Flag Assets

The public MVP uses local static SVG flag assets for consistent rendering across operating systems and browsers.

The current asset location is:

```text
frontend/public/flags/
```

The frontend should not rely on remote flag images at runtime.

When a team uses `flag_mode = "asset"`, the frontend must render the local image path stored in `flag_value`, such as `/flags/arg.svg`. ISO alpha-2 codes or FIFA-style codes may be fallbacks if an asset is unavailable or fails to load, but they must not replace valid local SVG assets.

Emoji flags may remain supported as a development or fallback representation, but they are not the primary public-MVP solution because some systems render emoji flags as regional letters.

## 13. Bracket Model

The tournament begins at the Round of 32.

The bracket contains:

- 16 Round-of-32 matches
- 8 Round-of-16 matches
- 4 quarter-final matches
- 2 semi-final matches
- 1 final match

The third-place match is not part of the MVP unless explicitly added later.

Each match must have:

- A stable match ID
- A round identifier
- Two team slots or previous-match winner references
- A calculated probability for each team when both teams are known
- A selected winner
- A selection source

The selection source should distinguish:

- Model-selected winner
- User-overridden winner
- Official locked winner, in a later version

---

## 14. Probability Models

The application must support a model abstraction.

### 13.1 MVP Model: Simple Elo

The MVP model is:

```text
simple_elo
```

This model converts rating difference directly into win/advancement probability.

### 13.2 Future Model: Historically Informed Elo

A later model is:

```text
historical_elo
```

This model starts from Elo/rating difference but applies calibration learned from historical match outcomes.

The historically informed model should be trained offline in Python and exported as static model parameters for the frontend.

### 13.3 Model Toggle

Once both models exist, the frontend must provide a toggle or selector between:

- Simple Elo
- Historically informed Elo

Changing the selected model must recompute:

- Match probabilities
- Predicted winners
- Bracket propagation
- Champion probabilities
- Top four champion summary

The model toggle must not require a backend.

---

## 15. Champion Probability Calculation

The public app should use exact probability propagation where possible, rather than requiring Monte Carlo simulation for every user interaction.

The reason is that exact propagation is:

- Faster
- Deterministic
- Easier to test
- Suitable for browser-side execution
- Suitable for static deployment
- Free of random sampling noise

Monte Carlo simulation may still be used in Python as a development, validation, or comparison tool.

Monte Carlo is not required for the deployed MVP.

---

## 16. Baseline Forecast

The baseline forecast is the default model forecast.

In the baseline forecast:

- No user overrides are applied.
- No manual scenario changes are applied.
- The selected model is used.
- The model selects the most likely winner of each match.
- Champion probabilities are calculated from the initial bracket and the selected probability model.

The baseline forecast represents the model’s default prediction.

---

## 17. Current Scenario Forecast

The current scenario forecast reflects the user’s manual changes.

Users may manually choose the winner of a match. When they do:

- The selected winner advances.
- Downstream matches are recalculated.
- Probabilities for affected future matches are recalculated.
- Champion probabilities are recalculated for the current scenario.

Manual overrides must be local to the user’s browser in the MVP.

Manual overrides should not modify the original data files.

---

## 18. Local Persistence

The MVP may support optional browser-side persistence.

If implemented, user selections may be stored locally using browser storage.

There will be:

- No user accounts
- No login
- No backend storage
- No shared database of user scenarios

Local persistence is optional for the MVP.

---

## 19. Shareable Scenarios

Shareable scenario URLs are not part of the MVP.

However, the architecture should not block them.

A later version may encode user overrides in the URL so that a user can share a modified bracket scenario with another person.

The data model should keep match IDs stable so that URL-encoded scenarios remain possible later.

---

## 20. Official Result Locking — Future Version

A later version should support official result locking.

This feature allows the project administrator to update the bracket as real World Cup matches are played.

A locked result means:

- The official winner advances.
- The locked result is treated as part of the baseline state.
- Downstream probabilities are recalculated from that new official state.
- The locked result cannot be changed by normal user interaction.
- The UI visually marks the match as an official completed result.

In the static-site architecture, this should be implemented by editing or generating a static results file and redeploying the website.

The MVP should not include a live admin panel.

---

## 21. Forecast History — Future Version

A later version should track forecast history across the tournament.

Every time official results are locked and the site is redeployed, the system should be able to save a forecast snapshot.

A forecast snapshot should record:

- Snapshot ID
- Timestamp
- Completed/locked matches at that time
- Champion probability for each remaining team
- Top four most likely champions
- Data source versions
- Rating snapshot used
- Model version used

This would allow the website to show how each country’s title probability changed over time.

### Version 1 — MVP Forecast Website

Version 1 includes:

- Static React + TypeScript frontend
- Local rating CSV
- Generated static JSON
- 32-team knockout bracket
- Local SVG flag assets
- Simple Elo probability model
- Baseline forecast
- Manual user overrides
- Current scenario forecast

- Historical rating-difference calibration
- Exported calibrated model parameters

- Model explanation text for users

Version 3 may include:

- Static SVG or PNG flag assets
- Better responsive bracket layout
- Shareable scenario URLs
- Improved visual design
- Better explanatory model notes
- More complete frontend testing

### Version 4 — Tournament Live Update Mode

Version 4 may include:

- Official result locking by the project administrator

## 23. Testing Priorities

The highest-priority tests are:

1. Probability math
2. Simple Elo rating-difference probability conversion
3. Historically informed Elo calibration, once implemented
4. Model toggle behavior, once implemented
5. Probability values summing to 1.0
6. Bracket propagation
7. Manual override behavior
8. Downstream recalculation after overrides
9. Champion probability calculation
10. Handling of missing teams or missing ratings
11. Stable match IDs
12. Data validation before frontend build

The project must not rely only on visual testing.

The mathematical and bracket logic must be covered by automated tests.

---

## 24. Non-Goals

The MVP will not include:

- Runtime backend
- Live Python server
- User accounts
- Login
- Backend database
- Betting odds
- Player-level predictions
- Injury modeling
- Weather modeling
- Group-stage prediction
- Real-time scraping during user interaction
- Live admin dashboard
- Paid server infrastructure

---

## 25. Architectural Rule for Coding Agents

Coding agents must preserve the following rule:

> The deployed public website must be able to run as a static site. Do not introduce backend runtime dependencies unless explicitly requested in a later architecture revision.

If a proposed implementation requires a server, database, authentication layer, or runtime Python process, it must be rejected for the MVP.

---

## 26. Summary

The project will use a professional React + TypeScript frontend and a Python-based offline data/model preparation workflow.

The public app will run as a static website.

All user-facing bracket recalculation will happen in the browser.

The MVP will focus on a transparent, testable simple Elo model and a polished bracket interface.

Future versions may add historically informed Elo, model toggling, shareable scenarios, official result locking, and forecast history across the tournament.
